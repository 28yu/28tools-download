## 実装上の重要な学び

### 1. 罫線が「fill された薄い rectangle」で描画される

構造図 PDF では罫線が `stroke` ではなく、**幅 0.5pt 程度の薄い filled rectangle** として描画されることが多い。`stroke` だけを拾う実装は取りこぼす。

```
constructPath [[19], [805.747, 1008.059, 0.459, 2.660], ...]
fill
```

→ `OPS.fill` / `OPS.eoFill` 後の最後の `constructPath` も処理対象に含める。

### 2. PathOps の内部コードは OPS テーブルとは別

`constructPath` の `args[0]` に格納される内部コード（PathOps）は、`OPS` 名前空間とは別系統:

| コード | 意味 |
|---|---|
| 13 | moveTo |
| 14 | lineTo |
| 15 | curveTo（6 引数） |
| 16 | curveTo2（4 引数） |
| 17 | curveTo3（4 引数） |
| 18 | closePath |
| 19 | rectangle（4 引数: x, y, w, h） |

これは `pdfjs-dist` の TypeScript 定義には型としては公開されていないので、ソース (`src/shared/util.ts` の `PathOps`) 参照。

### 3. ページ回転と座標変換

PDF2 p.1 は `page.rotate=90` で、MediaBox は 842×1191（縦）だが、表示は 1191×842（横）。

- `operatorList` の座標は **MediaBox 座標**
- 表示・ユーザー座標に変換するには `page.getViewport().transform` を適用する必要がある
- 加えて、各 op の CTM スタック (`save`/`restore`/`transform`) を追跡してからの最終変換

### 4. PDF2 p.1 のレイアウトは「カード集合」

最も重要な学び。当初の想定（Excel風の uniform table）と異なり、実際は:

```
[行ラベル列 | FG1カード | FG2カード | FG3カード | ... ]
            ├ 断面図   ├ 断面図   ├ 断面図
            ├ 主筋径   ├ 主筋径   ├ 主筋径
            ├ a|中央|b ├ a|中央|b ├ a|中央|b|c|d (連スパン)
            └ ...     └ ...     └ ...
```

- 各 FG は**独立した小カード**で、カード内に「位置 × 配筋プロパティ」のサブテーブルを持つ
- 連スパン梁（例: FG17,18,(19)）はカード幅が大きく、内部列数も変わる
- カード横断で共通する罫線（行ラベル列の境界、最上段の断面図ベースライン）と、カード固有の罫線（位置区切り）が混在する

→ 単純な「全 H 線をクラスタ → 全 V 線をクラスタ → グリッドにする」だけでは破綻する。次フェーズで **「カード境界検出」ステップ** が必要。

## 次フェーズに向けた方針修正

### 改訂アーキテクチャ案

```
1. 罫線抽出（本検証で完了）
   ↓
2. カード境界検出  ← 新規ステップ
   - 外枠 rectangle を見つける（PDF が rect として明示しているか、ヒューリスティックで矩形領域を判定）
   - または「行ラベル列」を特定し、横方向に均等間隔で繰り返される領域パターンを認識
   ↓
3. カードごとに内部グリッド構築
   - カード内に閉じた行・列クラスタリング
   - 行ラベル列のテキスト（主筋径/主筋上/主筋下/端あき/...）を取得
   - 位置列のラベル取得（a/中央/b/...）
   ↓
4. スキーマ列にマップ
   - 行ラベル + 位置ラベル → スキーマ v1 の `x_主筋径` / `x_上_1段` 等
   - パース後と原文の両方を出力
   ↓
5. プレビュー UI でユーザー確認・修正
   ↓
6. SheetJS で .xlsx 生成
```

### 次に取り組むタスク（優先順）

1. **カード境界検出ロジック**（最重要）
   - PDF2 p.1 の各 FG カードの外枠 bounding box を自動検出する
   - 出力: 各カードの (xMin, yMin, xMax, yMax) リスト
2. **カード内サブグリッド構築**
   - 行ラベル列（最左）のテキスト「主筋径」「主筋上」「主筋下」「あばら筋」「端あき」を取得
   - 位置列のラベル（a/中央/b/...）を取得
   - 行 × 位置のセルを構築
3. **配筋パーサー実装**（スキーマ v1 仕様）
   - `D25` → `主筋径=D25`
   - `6+2` → `1段=6, 2段=2`
   - `D13[2]@150` → `あばら径=D13, 脚数=2, ピッチ=150`
4. **S 梁ページ (PDF2 p.2) の検証**
   - レイアウトが RC 梁と異なる（カード型ではなく行型かもしれない）ため別途検証必要
5. **OCR フォールバック検証**（PDF1 全壊ページ用、Step 3）

## 方針修正（2026-05-20 追記）

レポート初版を書いた直後、設計哲学上の重要な誤りに気付いたため記録する。

### 誤っていた発想

「PDF の表構造を Excel に再現する」を前提とし、その延長で **「カード境界検出 → 内部グリッド構築 → セル位置からスキーマ列にマップ」** という実装方針を立てていた。

### 正しい発想

Excel は **Revit ファミリパラメータの中間表現** であり、PDF レイアウトの再現が目的ではない (詳細は `PDF-to-Excel-Schema-v1.md` 冒頭の「設計哲学」を参照)。

抽出側に必要なのは「梁ごとの値を拾うこと」だけで、PDF の表構造を再現する必要はない。罫線駆動で「セルの完全再現」を目指す方向は **過剰** で、より軽量な **アンカーテキスト駆動抽出** で済む。

### 新方針: アンカーテキスト駆動抽出

```
1. テキスト要素の位置取得（pdfjs getTextContent + 座標変換）
   ↓
2. アンカーテキスト識別
   - 符号アンカー: "FG1", "FG2", ..., "FG19" → 梁の起点
   - 行ラベルアンカー: "主筋径", "主筋上", "主筋下", "あばら筋", "端あき", "位置"
   ↓
3. 各 FG アンカーごとに:
   - X 方向: 隣の FG アンカーまでの範囲を「この梁の領域」とする
   - Y 方向: 行ラベルアンカーの Y 座標と一致するテキストを、その行の値とする
   - X 方向: 位置ラベル ("a", "中央", "b" 等) または等間隔区切りで a/b/c/d/e に振り分ける
   ↓
4. スキーマ v1 の列にマップ → 1 梁 1 行で出力
```

罫線抽出は **不要** だが、テキスト位置が極めて密で衝突する場合の補助としては使える（バックアップ手段）。

### Step 2 プロトタイプの位置付け

- `extract.mjs` / `render.mjs` / `grid.mjs` は **「罫線抽出という手段が機能する」ことの実証** として価値があり残す
- ただし本実装は別パス（`pdf-to-excel/src/`）で **アンカーテキスト駆動** で再実装する
- カード境界検出は実装しない

### 次にやるべきこと（更新版）

1. **アンカーテキスト駆動抽出器の試作**（PDF2 p.1 RC 梁を題材）
   - FG1〜FG19 の符号位置を取得
   - 行ラベル「主筋径」「主筋上」「主筋下」「あばら筋」「端あき」の Y 座標を取得
   - 各 FG ごとに位置 a/b/c/d/e × 各行ラベル のセル値を収集
   - スキーマ v1 形式で JSON 出力
2. **S 梁ページ (PDF2 p.2) で同方式検証**
   - 1 行 1 梁の単純レイアウトなので、よりシンプルになる想定
3. **OCR フォールバック検証**（PDF1 全壊ページ用、Step 3）
4. **配筋パーサー実装**（`D25` `6+2` `D13[2]@150` の構文解析）
