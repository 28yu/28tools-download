# PDF → Excel ツール — 実装イテレーション履歴

このドキュメントは「**なぜそうしたか**」の判断記録です。コードを見ても
背景が分からない事項を書き残します。コミットメッセージと併せて参照。

## v1: 初期 Node プロトタイプ (`pdf-to-excel/src/`)

- pdfjs-dist でテキスト抽出
- アンカーテキスト方式 (FG/SG 等の符号を基点に固定オフセットで値を拾う)
- 4 種類の抽出器 (RC基礎大梁/S大梁/柱OCR/小梁OCR)
- ExcelJS で .xlsx 出力 (Meta + 4シート)
- **判断**: アンカー方式を選んだのは、構造図リストが**符号(FG1, SG1, SC1)を基準とした規則的なグリッド配置**であるため。表形式の解析より堅牢。

## v2: スキーマv1 確定 + UI/UX 反復

ユーザフィードバックで Excel 構造を反復:
1. 構造マテリアルの追加 → ファミリの「構造マテリアル」パラメータと直結
2. 位置 a/b/c を**左端/中央/右端**に置換 → 図面慣習
3. シート名に構造種別を併記 → `基礎大梁 (RC)`, `大梁 (S)`
4. 3 行ヘッダで 主筋/あばら筋 をグループ化 → 視認性
5. 小梁の符号を小文字 (`sb19`) → JIS慣習
6. `_mm` サフィックス削除 → 全列mm前提なので不要

**判断**: スキーマは「Revit ファミリパラメータと 1:1 マッピングできる」ことを優先。1セル=1パラメータ。

## v3: ブラウザ移植 (`pdf-to-excel/web/`)

- 配布形態として**完全ブラウザ完結**を採用
- pdfjs-dist / ExcelJS / Tesseract.js を CDN 経由でロード
- Tesseract.js は重い (~16-40MB) ので**遅延ロード**
- SHA-256 + Cache API で同一PDFの再アップロードを高速化

**判断**: サーバを持たないことで:
- ユーザ機密 (構造図) を外部送信しない (法務的安心)
- インフラコストゼロ → 個人開発で無料配布可能
- 28tools.com の GitHub Pages にそのままホストできる

## v4: 28tools.com デザインシステム適用

`css/style.css` の `--concrete-gray`, `--blueprint-blue`, `--architect-green` 等
を流用。Inter + Noto Sans JP フォント。フッターを他ページと同構造に。

**判断**: 配布サイトとして本家と一貫した UX が必要。

## v5-v8: OCR の試行錯誤

### 試したこと
- DPI 300 → 400 → 500 → 600 と段階的に上げる
- 自前の grayscale + threshold 前処理
- PSM 11 / 6 / 3 の 3 パスマージ
- `_best` モデル切替
- `user_defined_dpi` パラメータ

### 学び
- **`_best` モデルが最大の精度向上要因** (~30%)
- **`user_defined_dpi` を設定しないと Tesseract が低 DPI と誤推定して文字を捨てる** — これが「文字が見つからない」最大の原因だった (`Estimating resolution as 406` 警告)
- **自前前処理は逆効果** — Tesseract 内蔵処理に任せる
- **3 PSM パスは効く** — 単一PSMの取りこぼしを補い合う

判明後の最終構成:
```
DPI 600 描画 + 前処理なし + jpn_best + jpn+eng + 3 PSM + user_defined_dpi=600
```

## v9: マテリアル抽出のトークン化

### 発見した重大バグ
古いコード: 同じ y 座標の単語を**スペースなしで連結**してから regex マッチ。
結果として `SS400` `400` (隣接セル) が `SS400400` になり、貪欲マッチで誤検出。

ログ表示:
```
🔍 P3 OCR マテリアル検出: SS400400, SS400166060, SS40012206060T, ...
```

### 解決
**文字幅 + 隣接距離**でトークン化:
- pdf.js では gap < 2pt → 同じ単語の続き
- OCR では gap < 10px → 同じ単語の続き
- それ以上の gap → 別単語

これで分割語 (`SN` + `490B` → `SN490B`) と隣接セル (`SS400` + `400` → 別物) が
正しく区別される。

### 鋼材グレード優先順位
出現回数で多数決すると、ノイズが多い `SS400` が勝つことがあるため、
JIS慣習で構造用優先:
```
SN > SM > BCP > BCR > STKR > STK > SS
```

## v10: JIS H 形鋼カタログ補完 (画期的)

### 動機
OCR が小梁の section を高頻度で truncation:
- `H-200×1` (本来 ×100×5.5×8)
- `H-298×1`
- `H-346×1`
- `H-300×150×65×9` (×6.5×9)

### 解決策
**H 形鋼は規格品**なので height H が分かれば残りが一意に決まる:
```
H = 200 → 自動的に B=100, tw=5.5, tf=8
H = 298 → B=149, tw=5.5, tf=8
H = 346 → B=174, tw=6, tf=9
```

JIS G 3192 narrow-flange + wide-flange の合計 38 件を `JIS_H_SECTIONS` 配列に。

### Note
- 角形鋼管 (□) には**適用しない** — 同じ (A,B) で板厚バリエーションあり
- 組立 H (BH) には**適用しない** — カスタム製作品
- 抽出値に `_推定: true` フラグ → 原文に `(JIS推定)` 表示でユーザ確認可能

### 検証結果
小梁 50 件 PDF:
- 補完前: 1/50 完全抽出
- 補完後: **46/50 完全抽出 (92%)**

## v11-v12: 柱抽出の徹底改善

### 報告された不具合
ユーザ画面: 柱 1 件のみ、断面情報全空。

### 根本原因 (4つの独立バグ)
1. **Y 探索範囲が +600px に hard-coded** — 500 DPI では 86pt しかない (本来 1500-4000px 必要)
2. **detectOcrCategory で SC=2 と過少** — Tesseract.js の `(SC5)` 等の punctuation 付き label を `^...$` 厳密マッチで弾いていた
3. **anchor 抽出も同様の厳密マッチで取りこぼし**
4. **section regex に `^` アンカー** — `L1-400×400×22` (OCR が `□` を `L1` と誤読) を弾く

### 解決
1. `pageBottom` まで探索
2. detectOcrCategory を**部分マッチ + 単語境界**に変更
3. anchor 抽出も部分マッチ化、punctuation 自動剥がし (`(SC5)` → `SC5`)
4. section regex を非アンカー化、`[xX×]+` で重複×吸収、`[-=]?` で誤読対応

## v13: 行クラスタリング撤廃

### 問題
ユーザ報告: `OCR: 1408語 (SC=4, SB=0) → column` だが **柱 1 件しか抽出されない**。

### 原因
旧アルゴリズム「Y 行で集約 → アンカーが最多の行を canonical 行とする」が、
Tesseract.js best モデルが SC1-SC6 を**バラバラの y**で返すと破綻 (各行1個ずつ
で同点 → タイブレークで最初の1個 → 1柱)。

### 解決: 符号名グループ化
行クラスタリングを完全撤廃し、**ユニークな符号名**で集約:
- 同じ符号が複数 y 座標で出現する場合、**直下に section-like text が最多の y** を canonical position として採用
- ヘディング・凡例エリアの重複は自動的に rejected

これで「OCR が SC1-SC6 を縦に並べて返さない」場合でも 6 柱抽出可能に。

## v14: 別名規約への対応 (`C\d+`)

### 報告
新サンプル PDF (test_柱リスト2.pdf) は `C101, C102, C103, C104, C104A, C105`
の通り符号形式。前バージョンが `SC|CC|CFT` のみで `C\d+` は除外していた。

### 解決
`C\d+` を anchor として許容 + **左右両方の単語境界**で守る。
これで `BCR295` (R が左にある) の `C295` 部は依然弾かれ、`C101` は OK。

### 同時発見: 空キャッシュバグ
ユーザ画面に「**キャッシュから即座にロード**」と出ていたのに結果が空 → 過去の失敗抽出をキャッシュしていた。**0 件結果はキャッシュ保存しない**ように修正。

## v15: 大梁 OCR 抽出器 + マルチバンド対応

### 問題
画像PDF の大梁リスト (G101-G107A) で「OCR でも抽出できず」。

### 原因
**大梁用の OCR 抽出器が存在しなかった**。柱用と小梁用しか作っていなかった。

### 解決
- `extractLargeBeamsFromOcr()` を新規実装
- detectOcrCategory に `lg` (large-beam) カウントを追加
- anchor regex: `(CSG|SG|FG|BG|G)\d+[A-Za-z]?`

### 副次的発見: マルチバンドテーブル問題
G101 (y=805) と G104 (y=3543) が**両方 x=1469**。隣接ソートで境界が幅0に。
- `assignXCardBounds()` で x クラスタリング → 同じ x の anchor は同じ境界を共有
- 各 anchor の探索範囲を `anchor.y + 5` 〜 pageBottom に (旧: 全 anchor の topY)

### 副次的発見: B 連結エラー
`SH-700×25014×28` のような OCR 連結 (250+14 が 25014 に) を桁分割で補正:
```
B=25014 (>1500) → split → B=250, tw=14, 既存tw=28 を tf に移動
```

### 副次的発見: 小数点欠落 + H run-on
- `tw=55` → `5.5`, `tf=75` → `7.5` (10倍チェック)
- `H=1756` → `175` (末尾桁削除)

### 検証結果
| PDF | 修正前 | 修正後 |
|---|---|---|
| 大梁 G101-G107A (12件) | 0 件 | **12 件 (100% 寸法完全)** |

## v16: 棟プレフィックス (`KC`/`KG`/`KCG`) + 区切り記号バリエーション

### 報告
JR 東京総合病院 (管理棟Ⅰ期) サンプルで **柱 0 件・大梁 0 件**。
小梁ページでも 42 件中 7 件超で断面型が空欄。

### 原因 1: 棟プレフィックス未対応
複数棟ある建物では `K` (管理棟), `R` (research wing) 等の棟プレフィックスを
符号に冠する慣習がある:
- `KC1, KC1A, KC1B, KC2 ... KC8` (柱)
- `KG1 ... KG8, KCG1, KCG2, KSG11` (大梁)

旧 anchor regex `(SC|CC|CFT|C)` / `(CSG|SG|FG|BG|G)` は内側の `C1` / `G1` には
マッチするが、`before` キャラクタ (`K`) が英字なので**境界チェックで弾かれていた**。

### 解決 1
- 柱: `([KR]?)(SC|CC|CFT|C)\d+[A-Z]?` — K/R を任意プレフィックスとして許容
- 大梁: `([KR]?)(CSG|SG|FG|BG|CG|G)\d+[A-Za-z]?` — 同じく + `CG` (片持ち大梁) を追加
- `detectOcrCategory` も同様に更新

K/R に限定したのは他の頭文字 (B/S/T) が grade 名 (`BCR295`, `STKR400`, `SS400`)
と衝突するため。安全な単語境界判定を保ちつつ、よく見る慣習だけ受け入れる。

### 原因 2: 小梁の断面型が×記号バリエーションを許容しない
旧 secRe = `(?:BH|SH|H|...)-?\d{2,4}(?:[xX×]+\d+...)`。
Tesseract.js は font hinting の都合で `×` を `*` (アスタリスク) や `·` / `・`
(中黒) に置き換えることがある。これらは regex で弾かれていた。

さらに OCR が **× 記号を完全に欠落**させ、トークンを空白区切りで返すケース
(`["H", "446", "199", "8", "12"]`) では concat fallback で結合しても
`H446199812` となり ×なしで regex 失敗。

### 解決 2
1. 区切り記号セットを `[xX×*·・]+` に拡張 (secRe + parseSection の normalizeSepX)
2. **行トークン再構築**を 3 段目のフォールバックとして追加:
   - 同じ y 行から `H/BH/SH` プレフィックストークンを発見
   - 続く純数値トークンを最大 4 個まで収集
   - 非数値・非区切り (`SS400`, `M16`) で停止
   - 正規化された `H-446×199×8×12` 形式を返す → parseSection が処理

### 検証 (synthetic)
| 入力 | 結果 |
|---|---|
| `KC1` ... `KC8` (8件) | 柱 8 件抽出 ✓ |
| `KG1`, `KCG1`, `KCG2`, `KSG11` 等 | 大梁全件抽出 ✓ |
| `H 446 199 8 12` (×欠落) | sb44 = H-446×199×8×12 ✓ |
| `H-450·200·9·14` (中黒) | sb45 = H-450×200×9×14 ✓ |
| `H-600*200*11*17` (アスタリスク) | sb60 = H-600×200×11×17 ✓ |

## v17: A1 ページの canvas オーバーフロー + 例外メッセージ整備

### 報告
新サンプル (CAD から PDF 書き出し, A1 1684×2384pt) で
「P1: テキスト抽出 0件 (画像のみ) → OCR フォールバック」 → 「**エラー: undefined**」。

### 原因 1: テキストがアウトライン化
PDF 内の content stream を解析すると `Tj`/`TJ` (テキスト表示) オペレータが
**ゼロ**で、全文字が `m`/`l`/`c` (ベジエパス) で描画されていた。
出力時に「テキストをアウトライン化」または「フォントを埋め込まない」設定で
PDF 化したため、pdfjs は永久に 0 件しか返さない。
→ OCR フォールバックは正しい挙動。

### 原因 2: A1 ページの canvas サイズ超過
A1 ページ (1684×2384pt) を 600 DPI で render すると
14033×19866 ≈ **278M ピクセル** のキャンバスが必要。
ブラウザの canvas 上限は ~268M (16384²) でこれを超過。
- Chrome は canvas.width/height を 0 に黙ってクランプ
- pdfjs の `page.render(...)` が **`message` プロパティを持たない例外**を投げる
- catch 節の `err.message` が undefined → 「**エラー: undefined**」表示

### 解決 1: 適応 DPI
`renderPage()` を改修:
- `MAX_DIM = 16000` (1辺), `MAX_AREA = 200M` (面積)
- 上限超過時は scale を縮小し effectiveDpi を返す
- A1 ページは自動で ~483 DPI にダウンスケール
- A3 / Letter はそのまま 600 DPI 維持

### 解決 2: Tesseract DPI 同期
`getWorker()` が effective DPI を受け取り、ページごとに
`user_defined_dpi` を再設定。グリフサイズ判定がレンダリング DPI と整合。

### 解決 3: 例外メッセージのフォールバック
catch 節を:
```js
err?.message || (typeof err === 'string' ? err : null)
            || err?.name || err?.toString?.() || 'unknown error'
```
非 Error 系オブジェクト (DOM Event, 文字列, plain dict) からも
意味のある文字列を取り出す。

### 検証結果
| 項目 | 修正前 | 修正後 |
|---|---|---|
| A1 サンプル PDF | エラー: undefined | OCR 完走 ~483 DPI |
| A3 / Letter | 600 DPI 維持 | 600 DPI 維持 |
| 例外メッセージ | "undefined" 出るケースあり | 必ず何らかの文字列 |

### 教訓
- ブラウザの **canvas サイズ上限**を常に意識: 268M area / 16384 dim
- `err.message` 直アクセスは禁忌 — pdfjs / Tesseract / Canvas API の
  どれもが非 Error を throw する可能性がある
- 「テキストアウトライン化」PDF は技術的に救済不可 (vector→OCR で頑張る) なので、
  上流 (CAD/Revit エクスポート設定) で予防するのが正解

## 設計判断の記録 (Why)

### なぜブラウザ完結なのか
- 構造図は**機密性が高い** (顧客情報、企業ノウハウ)。サーバ送信は法務リスク
- 個人開発で**インフラコストゼロ**にしたい
- 28tools.com (GitHub Pages 静的サイト) にそのまま乗せられる

### なぜ Tesseract.js なのか (他のOCR候補)
- **PaddleOCR**: 精度は高いが Web 配信モデルが大きい、ブラウザ完結が難しい
- **Cloud Vision API (Google/Azure)**: 高精度だが**サーバ送信必須** → 設計方針と矛盾
- **Tesseract.js**: 唯一のフルブラウザ OCR。精度は劣るが工夫で実用レベルに

### なぜ JIS カタログ補完なのか
構造設計者は規格品 (JIS H 形鋼) を 99% 使う。OCR で `H-298` まで読めれば、
あとは規格表で確定できる。**現実世界の制約を活用した補完**。

### なぜ「行クラスタリング」を捨てたのか
Tesseract.js の OCR 出力は予測不能で、同じ row のラベルが微妙に異なる y 座標
で返ってくる。Y で集約しようとすると壊れる → **意味 (符号名) でグループ化**
する方が頑健。

### なぜ「3 PSM パス」なのか
Tesseract の PSM (page segmentation mode) は文字配置の前提を変える。
- PSM 11 (sparse): 散らばったラベルに強い
- PSM 6 (block): 表形式テキストに強い
- PSM 3 (auto): 残りを拾う

単一 PSM では必ず取りこぼしが出るので 3 つマージ。時間は ~3倍 だがキャッシュで吸収。

## 設計ミスから学んだこと

### 1. キャッシュは絶対にバージョン管理
古い抽出ロジックの結果がキャッシュに残ると、新ロジックの改善が見えない。
ロジック変更時は必ず `CACHE_NAME` を bump。

### 2. 「空結果のキャッシュ」は害悪
「抽出 0 件」を成功結果として保存すると、修正後も 0 件が返り続ける。
**意味のある結果 (>0) だけキャッシュ**するのが原則。

### 3. ブラウザのキャッシュは複層
- Service Worker / Cache API
- HTTP cache
- アプリケーションキャッシュ

Ctrl+F5 だけでは不十分なケースあり。F12 → アプリケーション → サイトデータをすべてクリア が確実。

### 4. 「最多投票」型集約は危険
同点で全件タイの時に変な結果を出す。**ユニーク化 + スコアリング**の方が頑健。

### 5. OCR は確率的、regex は決定論的
Tesseract.js の出力は実行のたびに微妙に違うこともある。だから regex は**寛容に**書く。

## 移植可能なテクニック

このプロジェクトで蓄積した知見は他の OCR タスクに転用できる:

1. **JIS規格表バックフィル** — 規格品ベースのドメインでは強力
2. **anchor 符号でのクラスタリング** — 「行クラスタリング」より堅牢
3. **`user_defined_dpi` の明示** — どんな Tesseract.js プロジェクトでも有効
4. **DPI と PSM を独立に制御** — 多角的なカバレッジ
5. **`_推定` フラグ** — 補完値とOCR直値を区別してUIで明示
